What Broke When We Shipped Cron Jobs
(And What We Learned)

We built a cron system, a menu bar app, and 22 agents. Then we discovered that "it works on my machine" means nothing when the daemon runs at 3am and nobody's watching. Here are the real bugs and the real fixes.

The Silent Cron

We wired a CronService into the gateway daemon. It loaded jobs from cron.json, started a 60-second interval timer per job, and matched against cron expressions every tick. Clean code. Good tests. Shipped it.

Three days later: "Why aren't my crons firing?"

They were firing. The executor had zero logging. No console.log, no event emission, no trace. The daemon log showed "Cron started — 3 jobs scheduled" and then silence. For days.

// Before: invisible await cronService.start({ execute: async (agentId, message) => { const agent = agents.get(agentId); return agent.execute({ query: message }); }, }); // After: one line that saved hours execute: async (agentId, message) => { console.log(`🦖 Cron executing: agent=${agentId}`); // ... rest of executor }

Lesson: If a background process doesn't log its executions, it doesn't exist. The daemon's startup message was a lie — "3 jobs scheduled" told you nothing about whether they ever ran. Add execution logging before you ship, not after three days of "it works on my machine."

The Swift JSON Decoder Silent Failure

The menu bar app had a cron jobs view. It called the gateway's cron.list RPC, decoded the response into [CronJob], and displayed them. "No scheduled jobs" every time.

The gateway was returning 4 jobs. We verified with WebSocket. The data was there. The view was empty.

The culprit: try? JSONDecoder().decode(...)

// The gateway returns: { "id": "job_1", "name": "dream", "schedule": "0 3 * * *", "enabled": true, "lastRun": null, "nextRun": null } // The Swift model expects: struct CronJob: Codable { let command: String // ← missing from JSON! let lastRun: Date? // ← null in JSON, Date? can't decode null let lastResult: CronResult? // ← not in JSON at all } // Result: try? silently returns nil → empty array → "No scheduled jobs"

Three separate issues, all silenced by try?:

  1. Missing field: The gateway didn't include command (it stored it as message)
  2. Null dates: Swift's Date? decoder can't handle JSON null without a custom strategy
  3. Missing optional: lastResult wasn't in the payload at all

The fix was a manual JSON fallback parser that constructs CronJob from raw dictionaries when Codable fails. Not pretty, but it works with any payload shape.

Lesson: try? in Swift is a footgun for API responses. You're telling the compiler "if decoding fails, return nil and don't tell me why." Use try + catch and log the decode error, or write a lenient fallback parser from the start.

The RPC Name Mismatch

The menu bar app called cron.create. The gateway registered cron.add. "Server error."

The full mapping of mismatches:

Swift clientGatewayResult
cron.createcron.add❌ method not found
cron.deletecron.remove❌
cron.triggercron.run❌
cron.pausecron.enable(false)❌
cron.listcron.list✅

The gateway was written first, the Swift client second. Nobody shared a spec. The fix was registering aliases on the gateway so both names work.

Lesson: If you have a client and a server written in different languages, the RPC method names will diverge unless you generate them from a shared schema. An rpc-methods.json or OpenAPI spec is boring but prevents exactly this class of bug.

The Tool Call That Broke the Conversation

The Copilot API returned HTTP 400:

An assistant message with 'tool_calls' must be followed by tool messages responding to each 'tool_call_id'. The following tool_call_ids did not have response messages: call_z3zd7dhQ...

The LLM asked to call two tools. The first one threw an error. The for loop broke. The second tool never got a response message. The next API call had an assistant message with two tool_calls but only one tool response. 400.

The fix is embarrassingly simple:

for (const tc of response.tool_calls) { try { const result = await executeToolCall(tc); history.push({ role: 'tool', content: result, tool_call_id: tc.id }); } catch (err) { // ALWAYS push a response — even on error history.push({ role: 'tool', content: \`Error: \${err.message}\`, tool_call_id: tc.id }); } }

Lesson: When the LLM asks you to call N tools, you must respond to all N — even if some fail. The API contract doesn't care about your exceptions. We also added a sanitizer that strips orphaned tool_calls from assistant messages as a safety net.

The CheckedContinuation That Crashed on Wake

The menu bar app crashed every time the MacBook woke from sleep. The crash report pointed to _assertionFailure in CheckedContinuation.resume(returning:).

The bug was in a network port check that used NWConnection:

// The race: connection.stateUpdateHandler = { state in case .ready: continuation.resume(returning: true) // ← fires first case .cancelled: continuation.resume(returning: false) // ← fires AGAIN after cancel } // The timeout also calls cancel: DispatchQueue.asyncAfter(.now() + 1) { connection.cancel() } // cancel → .cancelled → resume() called twice → crash

Swift's CheckedContinuation will trap (crash) if you resume it more than once. This is by design — it's "checked." The fix was a boolean guard to ensure only the first resume goes through.

Lesson: Any time you combine NWConnection state handlers with timeouts, you have a double-resume race. Use a flag, or use withUnsafeContinuation (which doesn't trap on double-resume, though it's still UB).

The Version That Was Always 0.0.0

The UpdateAgent checked for new releases. It reported local: 0.0.0, latest: 0.0.0. Two bugs:

  1. require('fs') in an ESM module. The agent used CommonJS require to read package.json but the project compiles to ESM. Silent failure, fallback to 0.0.0.
  2. GitHub's /releases/latest returned the menu bar app release (v1.0.0-bar) instead of the framework release (v1.9.1). "Latest" means most recently published, not highest version.

Lesson: Don't mix require() and ESM import in the same project. And if you have multiple release tracks (framework + menu bar app), /releases/latest is the wrong endpoint — use /releases?per_page=10 and filter.

The Pattern

Every bug in this post has the same root cause: silent failure.

  • The cron executor didn't log → we didn't know it was running
  • Swift's try? swallowed decode errors → we didn't know why the list was empty
  • The RPC returned "method not found" → the UI said "server error" with no details
  • The tool call loop broke silently → the API complained 3 messages later
  • require('fs') failed silently in ESM → version was 0.0.0

The fix in every case was the same: make the failure loud. Add a log line. Replace try? with try + catch. Push an error response instead of breaking the loop. Use the import that actually works in your module system.

Background systems are especially dangerous because nobody's watching when they fail. If your daemon doesn't log every execution, your cron system is a prayer. If your decoder uses try? for API responses, your UI will lie to you.

Ship loud. Debug fast.
